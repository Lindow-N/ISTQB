2.1 Tester dans le SDLC

FL-2.1.1 — Impact du SDLC sur le test (K2)

Un SDLC = modèle définissant comment les phases de développement sont liées. Le choix du SDLC influe sur :

  • Périmètre & calendrier des activités de test
  • Niveau de détail de la documentation
  • Choix des techniques & approche de test
  • Degré d'automatisation
  • Rôle & responsabilités du testeur
ModèleImpact sur le test
Séquentiel (cascade, V)Revues d'exigences + analyse/conception tôt. Tests dynamiques tardifs (code à la fin).
Itératif/incrémental (spirale, Processus Unifié)Chaque itération = prototype fonctionnel → tests statiques & dynamiques à tous les niveaux. Régression poussée.
AgileDoc légère, automatisation poussée (régression), tests manuels souvent basés sur l'expérience.

FL-2.1.2 — Bonnes pratiques (tous SDLC) (K1)

  • À chaque activité de dev correspond une activité de test.
  • Les niveaux de test ont des objectifs spécifiques & différents (éviter redondance).
  • Analyse & conception commencent dès la phase de dev correspondante (test précoce).
  • Testeurs impliqués dans les revues dès que des versions préliminaires existent (shift left).
⚠️ PIÈGE D'INVERSION (vu en formation QSI)
QCM piège : « Pour chaque activité de test, il existe une activité de développement correspondante » → FAUX !
VRAI (syllabus) : Pour chaque activité de développement, il existe une activité de test correspondante.
Pourquoi ? Le test englobe le développement — il suit TOUTES les phases de dev, et ajoute ses propres activités (revues, analyse de risques, etc.). Le sens est : dev → test, jamais l'inverse.

FL-2.1.3 — Développement piloté par les tests (K1)

TDD

Tests écrits en premier, puis code pour les satisfaire, puis refactorisation. Dirige le codage.

ATDD

Tests dérivés des critères d'acceptation. Tests avant le développement de la fonctionnalité.

BDD

Comportement en langage naturel Given/When/Then, traduit en tests exécutables.

TDD, ATDD, BDD = approches shift-left (tests avant le code), itératives. Implémentent le principe du test précoce.

FL-2.1.4 — DevOps & tests (K2)

DevOps = dev (dont test) + exploitation travaillent ensemble. Changement culturel. Favorise CI/CD, feedback rapide.

Avantages ✅Défis / Risques ❌
Feedback rapide sur la qualitéPipeline à définir/établir
Shift left (code qualité + analyse statique)Outils CI/CD à introduire/maintenir
Automatisation CI/CD, env. stablesAuto des tests difficile à mettre en place
Vue sur le non-fonctionnel (perf, fiabilité)—
Moins de manuel répétitif, moins de régression—
DevOps = beaucoup d'automatisation, mais les tests manuels (surtout point de vue utilisateur) restent nécessaires.

FL-2.1.5 — Shift left (K2)

Shift left = tester plus tôt dans le SDLC (test précoce, §1.3). Ne signifie pas négliger les phases ultérieures !

  • Examiner la spécification du point de vue des tests (trouve ambiguïtés, incomplétudes).
  • Cas de test avant le code + exécution dans un harnais pendant l'implémentation.
  • Intégration continue / développement continu.
  • Tests statiques du code avant les tests dynamiques.
  • Tests non fonctionnels dès le niveau composant si possible.
Coût/effort supérieur au début, mais économies en fin de processus. Il faut l'adhésion des parties prenantes.

FL-2.1.6 — Rétrospectives (K2)

À la fin d'un projet/itération/jalon de livraison. Participants : testeurs, devs, architectes, PO, analystes métier.

  • 3 questions : Succès à conserver ? / Améliorations possibles ? / Comment intégrer ?
  • Résultats enregistrés → partie du rapport de clôture des tests.
  • Essentielles pour l'amélioration continue (les recommandations doivent être suivies).
🧠 Bénéfices pour les tests
Efficacité/efficience ↑ · Qualité du testware ↑ · Cohésion d'équipe · Qualité de la base de test ↑ · Coopération dev/test ↑.

2.2 Niveaux & types de test

FL-2.2.1 — Les 5 niveaux de test (K2)

Groupes d'activités organisées et gérées ensemble. Chaque niveau = une instance du processus de test. Distinction par : objet, objectifs, base de test, défauts/défaillances, approche/responsabilités.

1

Composants (unitaires)

Composants isolés. Souvent par les développeurs, dans leur environnement. Harnais/frameworks.

2

Intégration composants

Interfaces & interactions entre composants. Stratégie : ascendante, descendante, big-bang.

3

Système

Comportement/capacités du système complet. Fonctionnel, bout-en-bout, non fonctionnel. Équipe indépendante.

4

Intégration système

Interfaces avec systèmes/services externes. Env. similaire à l'opérationnel.

5

Acceptation

Validation & aptitude au déploiement. Idéalement par les utilisateurs. UAT, opérationnelle, contractuelle/réglementaire, alpha, bêta.

FL-2.2.2 — Les 4 types de test (K2)

📋 Test fonctionnel

« Quoi » (les fonctions). Vérifie complétude, exactitude, adéquation. Basé sur spécifications.

⚙️ Test non fonctionnel

« Comment » (comportement). Caractéristiques qualité ISO 25010 : performance, compatibilité, utilisabilité, fiabilité, sécurité, maintenabilité, portabilité.

⬛ Test boîte noire

Basé sur les spécifications (doc externe). Vérifie le comportement vs spécifs.

⬜ Test boîte blanche

Basé sur la structure interne (code, architecture, flux). Vise la couverture de la structure.

Les 4 types peuvent s'appliquer à TOUS les niveaux de test (mais l'accent diffère).
⚠️ Piège fonctionnel vs non-fonctionnel
Un test non fonctionnel peut dériver d'un fonctionnel : même fonction, mais on vérifie une contrainte (ex : s'exécute en moins de 2 s, portable sur une nouvelle plateforme).

FL-2.2.3 — Confirmation vs Régression (K2)

Test de confirmationTest de régression
Confirme que le défaut original est corrigéConfirme aucune conséquence négative d'une modif
Rejoue les cas qui avaient échoué + nouveaux testsVérifie le composant modifié ET les autres, voire d'autres systèmes
—Commencer par une analyse d'impact
—Suites rejouées souvent → bons candidats à l'automatisation
Confirmation + régression sont nécessaires à tous les niveaux si défauts corrigés ou modifications.

2.3 Tests de maintenance

FL-2.3.1 — Tests de maintenance & déclencheurs (K2)

Système en production. Maintenance = corrective, adaptative, perfective. Inclut livraisons planifiées et correctifs à chaud (non planifiés).

Analyse d'impact avant modification → décider si on change, au vu des conséquences. Périmètre du test dépend :

  • Degré de risque des changements
  • Taille du système existant
  • Ampleur des modifications

3 catégories de déclencheurs

DéclencheurExemples
ModificationsAméliorations planifiées, correctives, correctifs à chaud
Migrations/mises à niveauChangement de plateforme, conversion de données
RetraitFin de vie → tests d'archivage des données, procédures de restauration/extraction
Test de maintenance = évaluer le succès des changements + vérifier les régressions dans les parties inchangées.
