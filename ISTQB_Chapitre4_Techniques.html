4.1 Aperçu des techniques de test

Les techniques de test aident à développer de manière systématique un ensemble de cas de test restreint mais suffisant. Elles aident à définir les conditions, identifier les éléments de couverture & les données de test.

FL-4.1.1 — Les 3 catégories (K2)

⬛ Boîte noire

Basée sur le comportement spécifié, sans référence à la structure interne. Cas indépendants de l'implémentation.

⬜ Boîte blanche

Basée sur la structure interne & le traitement. Cas créés après la conception/implémentation.

🧠 Basée sur l'expérience

Utilise les connaissances & l'expérience du testeur. Complémentaire des deux autres.

Boîte noire = basée sur les spécifications · Boîte blanche = basée sur la structure. Les techniques basées sur l'expérience détectent des défauts qui échappent aux autres.

4.2 Techniques boîte noire (K3 ★)

FL-4.2.1 — Partitions d'équivalence (K3)

Diviser les données en partitions : tous les éléments d'une partition sont traités de la même manière. Si une valeur détecte un défaut, les autres de la même partition le détecteront aussi → 1 test par partition suffit.

  • Partitions valides (traitées) / invalides (ignorées/rejetées).
  • Peuvent être continues, discrètes, ordonnées ou non, finies ou infinies.
  • Ne se chevauchent pas & non vides.

Élément de couverture : les partitions. Couverture 100% : exercer chaque partition (valide + invalide) au moins une fois.

✏️ Exemple K3
Champ « âge » acceptant 18–65 ans.
Partitions : invalide [<18] · valide [18–65] · invalide [>65]
→ 3 cas de test minimum (1 par partition) : ex. 10, 30, 80.
Couverture Each Choice (multi-paramètres) : chaque partition de chaque ensemble doit être exercée au moins une fois. Ne prend pas en compte les combinaisons.

FL-4.2.2 — Analyse des valeurs limites (K3)

Base sur l'exercice des limites des partitions (où les devs font le plus d'erreurs). Uniquement pour partitions ordonnées.

TechniqueÉléments de couverture par limiteCouverture 100%
2 valeursLa valeur limite + sa voisine de la partition adjacenteToutes les valeurs limites exercées
3 valeursLa valeur limite + ses 2 voisinesToutes les limites + voisines exercées
⚠️ Piège classique
3 valeurs est plus rigoureuse que 2 valeurs. Ex : si « si (x ≤ 10) » est mal codé « si (x = 10) », la 2-valeurs (x=10, x=11) ne détecte pas le défaut, mais la 3-valeurs (x=9) le détecte.
✏️ Exemple K3 — Champ 18–65 ans
2 valeurs : 17, 18, 65, 66 → 4 cas
3 valeurs : 16, 17, 18, 64, 65, 66 → 6 cas
(les bornes + leurs voisines de chaque côté)

FL-4.2.3 — Test par tables de décisions (K3)

Teste comment les combinaisons de conditions aboutissent à des résultats différents. Efficace pour enregistrer une logique complexe (règles de gestion).

  • Lignes = conditions & actions. Colonnes = règles de décision (combinaison unique).
  • Entrée limitée : booléen (V/F). Entrée étendue : valeurs multiples (plages, partitions…).

Notation : V (vrai) · F (faux) · - (non pertinente) · N/A (irréalisable) · X (action a lieu).

Élément de couverture : colonnes avec combinaisons réalisables. Couverture 100% : exercer toutes ces colonnes.

⚠️ Piège exponentiel
Le nombre de règles croît de manière exponentielle avec le nombre de conditions. → Utiliser une table réduite ou une approche basée sur les risques.
La table de décision permet de trouver des lacunes ou contradictions dans les exigences.

FL-4.2.4 — Test de transition d'état (K3)

Modélise le comportement via états & transitions valides. Une transition est initiée par un événement, qualifié par une condition de garde, pouvant entraîner une action.

Syntaxe : événement [condition de garde] / action

La table d'états (lignes = états, colonnes = événements) montre explicitement les transitions non valides (cellules vides), contrairement au diagramme.

Les 3 critères de couverture

CritèreÉlément de couvertureForce
Tous les étatsLes étatsPlus faible
Transitions valides (0-switch)Transitions valides uniquesLe plus largement utilisé
Toutes les transitionsValides + non validesLe plus rigoureux (critique/sécurité)
🧠 Hiérarchie de force
Tous les états < Transitions valides < Toutes les transitions.
Transitions valides 100% garantit tous les états 100%. Ne tester qu'une transition non valide par cas = éviter le masquage de défauts.

4.3 Techniques boîte blanche (K2)

Basées sur la structure du code. Ce syllabus couvre 2 techniques liées au code.

FL-4.3.1 — Test des instructions (K2)

Élément de couverture : les instructions exécutables. Couverture 100% = toutes les instructions exécutées au moins une fois.

Couverture des instructions 100% ne garantit pas que toute la logique de décision est testée (les branches peuvent ne pas l'être). Ne détecte pas les défauts liés aux données (ex : division par zéro).

FL-4.3.2 — Test des branches (K2)

Une branche = transfert de contrôle entre 2 nœuds. Peut être inconditionnelle (code en ligne droite) ou conditionnelle (résultat de décision : if/then, switch/case, boucle).

Élément de couverture : les branches. Couverture 100% = toutes les branches exercées.

La couverture des branches englobe celle des instructions : 100% branches → 100% instructions (mais pas l'inverse).

FL-4.3.3 — Valeur des tests boîte blanche (K2)

  • Force : toute l'implémentation est prise en compte → détecte les défauts même si la spec est vague, obsolète ou incomplète.
  • Faiblesse : si une exigence n'est pas implémentée, le test boîte blanche ne détectera pas le défaut d'omission.
  • Fournit une mesure objective de la couverture (la boîte noire ne le permet pas).

4.4 Techniques basées sur l'expérience (K2)

FL-4.4.1 — Estimation d'erreurs (K2)

Anticiper les erreurs/défauts/défaillances à partir des connaissances du testeur : comportement passé de l'app, erreurs typiques des devs, défaillances d'applications similaires.

Domaines : entrée, sortie, logique, calcul, interfaces, données.

Attaques de fautes = approche méthodique : liste d'erreurs possibles → tests conçus pour identifier/exposer/provoquer les défauts associés.

FL-4.4.2 — Test exploratoire (K2)

Tests simultanément conçus, exécutés et évalués pendant que le testeur découvre l'objet de test.

  • Structuré via le test basé sur des sessions : temps défini + charte de test (objectifs) + débriefing.
  • Utile quand les specs sont peu nombreuses/inadéquates ou sous pression de temps.
  • Plus efficace si testeur expérimenté + connaissance du domaine + créativité.
  • Peut intégrer d'autres techniques (ex : partitions d'équivalence).

FL-4.4.3 — Test basé sur des checklists (K2)

Concevoir/exécuter des tests pour couvrir les conditions d'une checklist (basée sur l'expérience, l'importance utilisateur, les échecs connus).

  • Éléments souvent formulés en question, vérifiables séparément & directement.
  • Ne doit pas contenir : éléments vérifiables automatiquement, critères d'entrée/sortie, éléments trop généraux.
  • Doivent être régulièrement mises à jour (analyse des défauts), sans devenir trop longues.
Checklists de haut niveau → couverture potentiellement plus grande mais répétabilité moindre.

4.5 Approches basées sur la collaboration (K2/K3)

Contrairement aux autres techniques (détection de défauts), celles-ci se concentrent aussi sur l'évitement des défauts par la collaboration.

FL-4.5.1 — Rédaction collaborative de User Stories (K2)

Une User Story = caractéristique utile à l'utilisateur/acheteur. Les 3 C (Jeffries) :

CDescription
CarteLe support décrivant la US (carte d'index, entrée tableau…)
ConversationExplique comment le logiciel sera utilisé (doc ou verbale)
ConfirmationLes critères d'acceptation
Format US : En tant que [rôle], je veux [objectif], afin de [valeur métier]
INVEST — Une bonne US doit être : Indépendante, Négociable, Valeur, Estimable, Simple/Petite, Testable.

FL-4.5.2 — Critères d'acceptation (K2)

Conditions à remplir pour que la US soit acceptée par les parties prenantes. = Conditions de test à vérifier.

Usages : définir le périmètre · consensus · scénarios +/− · base des tests d'acceptation · planification/estimation.

2 formats principaux

Orienté-scénarioOrienté-règles
Given/When/Then (BDD, §2.1.3)Liste à puces ou tableau entrée-sortie

FL-4.5.3 — ATDD (K3)

Développement piloté par les tests d'acceptation : cas de test créés avant l'implémentation de la US, par des membres aux perspectives différentes (clients, devs, testeurs).

Le processus ATDD

ÉtapeAction
1. Atelier de spécificationUS & critères d'acceptation analysés, discutés, rédigés. Lacunes/ambiguïtés résolues.
2. Création des cas de testBasés sur les critères = exemples du fonctionnement. « Exemples » et « tests » = synonymes.
Ordre des tests : positifs d'abord (comportement correct), puis négatifs, puis non fonctionnels (perf, utilisabilité).
Les cas de test doivent couvrir toutes les caractéristiques de la US et ne pas aller au-delà. Aucun cas ne doit décrire les mêmes caractéristiques. → Avec un framework d'auto, ils deviennent des exigences exécutables.
