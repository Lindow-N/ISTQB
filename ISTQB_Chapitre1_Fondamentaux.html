1.1 Qu'est-ce que le test ?

Tester = ensemble d'activités pour découvrir les défauts et évaluer la qualité des artefacts (= objets de test).

⚠️ Piège majeur
Tester ≠ juste exécuter le code (analyse, conception, revues…).

Vérification vs Validation (★ moyen mnémo pour l'examen)

QuestionRéférence
Vérification« Est-ce qu'on construit le produit correctement ? »les exigences / spec (ce qui est écrit)
Validation« Est-ce qu'on construit le BON produit ? »les besoins utilisateurs (ce que l'utilisateur veut vraiment)
🎯 Astuce QCM
Si l'énoncé dit « exigences » ou « spécifications » → Vérification.
Si l'énoncé dit « besoins », « utilisateurs », « réel » → Validation.

📖 Définitions à connaître par cœur (K1/K2)

TermeDéfinition (à mémoriser)
Base de testL'ensemble des connaissances utilisées comme base pour l'analyse et la conception des tests (spécifications, US, UC…).
Condition de testUn aspect des bases de test qui est pertinent pour atteindre des objectifs de test spécifiques : le « Quoi tester ».
Oracle de testUne source pour déterminer le résultat attendu à comparer avec le résultat réel du système testé.
TestwareProduits d'activités réalisés au cours du processus de test et utilisés pour la planification, la conception, l'exécution, l'évaluation et le reporting des tests.
Cas de testEnsemble d'entrées, conditions d'exécution et résultats attendus (issu de la conception).
Objet de testLe composant/système à tester (composant, module, écran, système complet…).
🧠 Mémo — les 2 questions clés
Analyse = « Quoi tester ? » → Conditions de test.
Conception = « Comment tester ? » → Cas de test.

Dynamique vs Statique

DynamiqueStatique
Exécute le logicielN'exécute pas
Provoque des défaillancesConstate directement les défauts

Objectifs typiques du test (K1)

  • Évaluer les artefacts (exigences, User Stories, code…)
  • Provoquer défaillances / trouver défauts
  • Assurer la couverture requise
  • Réduire le risque de mauvaise qualité
  • Vérifier exigences + conformité (contrat, loi, réglementation)
  • Informations pour décisions éclairées
  • Confiance dans la qualité + validation (complet, comme attendu)
Réponse « prouver qu'il n'y a aucun défaut » → toujours FAUX.
⚠️ PIÈGE OFFERT PAR LE FORMATEUR (vu en formation QSI)
Exemple de QCM : « Parmi ces éléments, lequel est un objectif typique du test ? »
  • A. ❌ Valider que les exigences documentées sont respectées → devrait dire « Vérifier » (exigences = vérification)
  • B. ✅ Provoquer des défaillances et trouver des défauts
  • C. ❌ Détecter les erreurs et identifier les causes racines → ce n'est PAS un objectif typique du test
  • D. ❌ Vérifier que l'objet répond aux attentes de l'utilisateur → devrait dire « Valider » (utilisateurs = validation)
Règle d'or ISTQB : les verbes « Vérifier » et « Valider » ne sont JAMAIS interchangeables. Les QCM adorent les inverser.
Reflexe : regarde le mot qui suit → « exigences/spec/conforme/documenté » ⇒ VÉRIFIER | « utilisateurs/besoins/attendu/réel » ⇒ VALIDER.

Test vs Débogage (K2)

TestDébogage
Déclenche défaillances / constate défautsTrouve & élimine la cause
Activité de testActivité de développement

Débogage : Reproduction → Diagnostic (cause racine) → Correction. Puis test de confirmation (le correctif marche ?) + test de régression (rien cassé ailleurs).

1.2 Pourquoi tester ?

Test vs Assurance qualité (K1)

Test (= Contrôle qualité)Assurance qualité
ApprocheCorrectivePréventive
FocusProduitProcessus
ResponsableÉquipe testTous
⚠️ Piège n°1 de l'examen : Test vs AQ vs Contrôle Qualité
« Test » et « Assurance qualité » ne sont pas synonymes. Le test est une forme de contrôle qualité.
  • « corrective, produit » → Contrôle Qualité / Test
  • « préventive, processus, tous » → Assurance Qualité
🧠 Moyen mnémo : Assurance Qualité = Avant (préventif) + Processus → Tous.  |  Contrôle Qualité = Corrige + Produit → Testeurs.

⚠️ À retenir
  • Un défaut ne provoque pas toujours une défaillance.

  • Analyse des causes racines = pour éviter la récurrence.

1.3 Les 7 principes du test (K2 ★ par cœur)

1

Présence, pas absence

Le test montre des défauts présents, jamais leur absence.

2

Exhaustif impossible

Tout tester = impossible. Prioriser via techniques + risques.

3

Tester tôt = économiser

Plus tôt on trouve un défaut, moins il coûte. Shift-left.

4

Regroupement

Quelques composants concentrent la plupart des défauts (Pareto).

5

Usure des tests

Répéter = inefficace. Sauf régression automatisée.

6

Dépend du contexte

Pas d'approche universelle.

7

Illusion 0 défaut

Corriger tous les défauts ≠ succès. Il faut aussi valider.

⚠️ Nuance d'examen : principe 1 vs 7
1 = on ne prouve pas l'absence de défauts. 7 = même sans défaut, on peut rater les besoins utilisateurs.

1.5 Compétences & bonnes pratiques

Compétences génériques (K2)

  • Connaissance test · Rigueur/curiosité · Communication · Esprit analytique/critique
  • Connaissances techniques · Connaissance du domaine
Efficacité = bonnes techniques (bien choisir). Efficience = outils/automatisation (économiser les ressources).
Le testeur = porteur de mauvaises nouvelles → communiquer les défauts de façon constructive (méfier du biais de confirmation).

Équipe intégrée (K1) — issue de XP

  • Tout membre compétent peut faire toute tâche · Chacun responsable de la qualité
  • Espace partagé · meilleure communication & synergie
  • Mais pas toujours adapté (ex. critiques pour la sécurité → indépendance requise)
